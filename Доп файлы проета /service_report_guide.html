Пояснение проекта простыми словами

Как работает сервис агрегации внеучебной деятельности университета

Этот документ нужен как понятная шпаргалка для защиты и отчета. Он объясняет, что делает сервис, какие роли в нем есть, как двигаются данные, какой стек используется и за что отвечают основные файлы проекта. Текст написан без сложных терминов, чтобы им можно было пользоваться даже без глубокого опыта в разработке.

FastAPI Backend-сервис с REST API и Swagger-документацией.
PostgreSQL База данных для пользователей, событий, категорий и записей.
HTML/CSS/JS Frontend-интерфейс, который открывается в браузере.
Docker Упрощает запуск проекта перед демонстрацией.

1. Что это за сервис

Сервис объединяет всю внеучебную активность университета в одном месте. Организаторы публикуют мероприятия: набор в клуб, секцию, концерт, подкаст, волонтерский проект, мастер-класс и так далее. Студенты заходят на общую доску, читают описание события, смотрят дату, место, контакты, обложку и при желании записываются.

Главная идея проекта: убрать хаос из разных чатов и объявлений и сделать один понятный цифровой сервис, где студенты быстро находят активности, а организаторы получают удобный инструмент публикации и учета участников.

2. Кто пользуется системой

Студент

Смотрит общую ленту событий, фильтрует мероприятия, записывается и видит свои записи в профиле.

Организатор

Создает и редактирует события, добавляет обложку, публикует или архивирует мероприятие и смотрит участников.

Администратор

Управляет пользователями и ролями, видит все события в системе и может контролировать их статусы.

3. Как пользователь видит сервис

Шаг 1. Главная страница. Даже без входа пользователь видит список актуальных мероприятий. Это удобно для первого знакомства с сервисом.
Шаг 2. Вход или регистрация. Чтобы записываться на мероприятия или создавать свои события, нужно авторизоваться.
Шаг 3. Работа по роли. Студент получает доступ к профилю и записи, организатор к созданию события, администратор к панели управления.
Шаг 4. Обмен данными с backend. Все действия интерфейса отправляют запросы на API FastAPI, а backend уже читает или меняет данные в PostgreSQL.

4. Что умеет студент

  1. Открыть главную страницу и посмотреть список мероприятий.
  2. Воспользоваться поиском и фильтром по категориям.
  3. Открыть карточку события и узнать, что это за мероприятие.
  4. Войти в аккаунт и нажать кнопку записи.
  5. Перейти в профиль и увидеть список своих записей.
  6. Отменить запись, если событие больше не интересно.

5. Что умеет организатор

  1. Создать новое событие через специальную форму.
  2. Заполнить название, краткое и полное описание, дату, место, контакты и формат.
  3. Выбрать категорию и при необходимости лимит участников.
  4. Загрузить изображение-обложку события.
  5. Сохранить событие как черновик или сразу опубликовать.
  6. Редактировать, архивировать или удалять свои события.
  7. Открыть список участников и увидеть ФИО, телефон и email записавшихся.

6. Что умеет администратор

  1. Видеть список всех пользователей системы.
  2. Менять роли пользователей: студент, организатор, администратор.
  3. Видеть все мероприятия в системе, а не только свои.
  4. Управлять событиями: редактировать, публиковать, отправлять в архив, удалять.
  5. Использовать сервис как человек, который контролирует корректность работы всей платформы.

7. Как работает backend

Backend написан на FastAPI. Это серверная часть проекта. Она принимает запросы от браузера, проверяет данные, работает с базой данных и возвращает результат обратно.

Пример. Студент нажимает кнопку "Записаться". Frontend отправляет запрос на backend. Backend проверяет: существует ли событие, опубликовано ли оно, есть ли еще места и не записан ли студент уже раньше. Если все в порядке, в базе создается запись.
Еще пример. Организатор создает событие. Backend получает форму, валидирует поля, сохраняет изображение обложки и записывает данные о событии в таблицу events.

8. Как работает frontend

Frontend реализован как браузерный интерфейс на обычных HTML, CSS и JavaScript. Он не хранит данные сам по себе, а только показывает информацию, которую получает от backend.

  1. index.html описывает структуру экранов: главная, вход, регистрация, профиль, создание события, админка.
  2. styles.css отвечает за внешний вид интерфейса: цвета, отступы, блоки, адаптивность под разные экраны.
  3. app.js содержит логику интерфейса: отправку запросов на API, переключение экранов, рендер карточек событий и работу кнопок.

9. Используемый стек технологий

Технология Где используется Зачем нужна
FastAPI Backend Создание REST API, обработка запросов, генерация Swagger-документации.
SQLAlchemy Backend Работа с PostgreSQL через Python-модели.
Pydantic Backend Проверка входных данных и валидация схем запросов и ответов.
PostgreSQL База данных Хранение пользователей, событий, категорий и записей.
Alembic Backend Миграции базы данных, чтобы структура таблиц создавалась и обновлялась правильно.
JWT Backend + Frontend Авторизация пользователя через токен после входа в систему.
HTML/CSS/JavaScript Frontend Формирование пользовательского интерфейса.
Nginx Frontend-контейнер Раздача статических файлов интерфейса.
Docker / Docker Compose Развертывание Быстрый запуск сервиса перед демонстрацией.

10. Основные сущности базы данных

Пользователь

Хранит ФИО, email, пароль в зашифрованном виде, факультет, группу, телефон и роль.

Событие

Содержит название, описание, дату, место, контакты, формат, статус, лимит участников и обложку.

Категория

Нужна для группировки мероприятий, например: концерты, клубы, секции, волонтерство.

Запись на событие

Связывает конкретного студента и конкретное событие. Позволяет понять, кто записался.

11. Логика работы по файлам

Файл Назначение
backend/app/main.py Точка запуска backend. Создает приложение FastAPI, подключает роуты, CORS и папку с изображениями.
backend/app/core/config.py Хранит настройки проекта: имя сервиса, путь API, параметры PostgreSQL, токены и папку загрузок.
backend/app/models/*.py Модели таблиц базы данных: пользователи, события, категории, регистрации.
backend/app/schemas/*.py Схемы Pydantic. Проверяют, какие данные можно принять и в каком виде отдавать ответы.
backend/app/api/v1/auth.py Регистрация, вход и получение данных текущего пользователя.
backend/app/api/v1/events.py Получение событий, создание, редактирование, изменение статуса, удаление, загрузка обложек.
backend/app/api/v1/registrations.py Запись на событие, отмена записи и просмотр участников.
backend/app/api/v1/users.py Профиль пользователя, его события, его записи, список пользователей и смена ролей.
backend/app/services/*.py Бизнес-логика. Здесь принимаются решения: можно ли записаться, можно ли удалить событие, есть ли права доступа.
backend/app/utils/security.py Шифрование паролей и работа с JWT-токенами.
backend/app/utils/files.py Сохранение загруженных изображений событий.
backend/alembic/versions/*.py Миграции базы данных. Создают структуру таблиц и новые поля.
frontend/index.html Главный HTML интерфейса. В нем описаны экраны и шаблон карточки события.
frontend/styles.css Внешний вид интерфейса: сетки, кнопки, формы, карточки и адаптация под мобильные устройства.
frontend/app.js Клиентская логика: отправка запросов на API, отрисовка событий, переключение экранов, работа роли пользователя.
backend/docker-compose.yml Файл запуска проекта через Docker Compose.

12. Как проходит типичный запрос внутри системы

1. Пользователь нажимает кнопку во frontend, например "Войти" или "Записаться".
2. Файл frontend/app.js отправляет HTTP-запрос на backend по адресу /api/v1/....
3. FastAPI принимает запрос в одном из роутов, например в auth.py, events.py или registrations.py.
4. Backend проверяет права доступа, валидирует данные и вызывает нужную бизнес-логику из папки services.
5. Сервис работает с PostgreSQL через SQLAlchemy-модели и получает или изменяет данные.
6. Backend отправляет готовый JSON-ответ обратно во frontend.
7. Frontend обновляет экран: показывает событие, уведомление, список участников или изменения профиля.

13. Что важно сказать в отчете

  1. Проект решает проблему разрозненности информации о внеучебных мероприятиях.
  2. Сервис поддерживает несколько ролей и разграничение прав доступа.
  3. Используется клиент-серверная архитектура.
  4. Backend и frontend разделены, поэтому систему можно дальше масштабировать.
  5. Применяется PostgreSQL как надежная реляционная база данных.
  6. Для удобства демонстрации и развертывания используется Docker.

14. Что можно назвать преимуществами проекта

  1. Одна точка входа для всех университетских мероприятий.
  2. Понятные роли: студент, организатор, администратор.
  3. Поддержка публикации событий с изображением и лимитом мест.
  4. Просмотр участников организатором.
  5. Русифицированная Swagger-документация API.
  6. Работа сервиса и на компьютере, и на телефоне по локальной сети.

15. Как кратко объяснить работу проекта руководителю

Сервис состоит из двух частей. Первая часть — backend на FastAPI, который хранит правила работы системы и общается с базой PostgreSQL. Вторая часть — frontend, который показывает пользователю интерфейс в браузере. Когда пользователь что-то делает, frontend отправляет запрос на backend, backend проверяет данные, работает с базой и возвращает результат. Благодаря этому организаторы могут публиковать мероприятия, студенты записываться на них, а администратор управлять пользователями и событиями.

Готово для использования в отчете. Этот файл можно открывать как обычную веб-страницу и использовать как основу для разделов: "Назначение системы", "Архитектура", "Выбор технологий", "Описание модулей", "Сценарии работы пользователя" и "Описание базы данных".
