План создания веб-сервиса агрегации внеучебной деятельности университета

1. Цель сервиса

Создать единую университетскую платформу, где:
- организаторы публикуют внеучебные активности;
- студенты просматривают общую доску мероприятий;
- студенты получают полную информацию о событии;
- студенты записываются на участие;
- университет получает централизованный канал коммуникации.

2. Основные пользователи

Студент:
- просматривает список событий;
- фильтрует и ищет мероприятия;
- открывает карточку события;
- записывается / отменяет запись;
- видит дату, место, контакты, описание.

Организатор:
- создает и редактирует события;
- указывает формат, лимит мест, контакты, место проведения;
- видит список записавшихся;
- управляет статусом события.

Администратор:
- модерирует контент;
- управляет ролями;
- контролирует категории, теги, жалобы, статистику.

3. Проблема, которую решает сервис

Сейчас внеучебная активность часто разрознена:
- объявления публикуются в разных чатах и соцсетях;
- студентам сложно найти актуальные мероприятия;
- организаторам трудно собирать участников;
- университету сложно вести учет активности.

Сервис решает это через единую доску мероприятий.

4. Ключевой функционал MVP

Минимальная версия должна включать:

1) Регистрация и вход
- авторизация по email;
- желательно через университетскую почту;
- роли: студент, организатор, администратор.

2) Общая доска событий
- список всех актуальных мероприятий;
- карточки с краткой информацией;
- сортировка по дате публикации и дате проведения.

3) Карточка события
- название;
- краткое и полное описание;
- категория;
- дата и время;
- место проведения;
- формат: офлайн / онлайн / смешанный;
- контакты организатора;
- число мест;
- количество записавшихся;
- кнопка записи.

4) Создание события организатором
- форма публикации;
- загрузка обложки;
- указание лимита участников или безлимитного посещения;
- статус события: черновик / опубликовано / завершено.

5) Запись студентов
- запись в один клик;
- отмена записи;
- подтверждение успешной регистрации.

6) Личный кабинет
- студент: мои записи;
- организатор: мои события и участники.

7) Админ-панель
- просмотр всех событий;
- скрытие или удаление неподходящих публикаций;
- управление категориями.

5. Типы событий, которые нужно поддержать

Сервис должен учитывать разные форматы:
- набор в секцию;
- набор в клуб или студенческое объединение;
- приглашение на концерт;
- запись на подкаст;
- волонтерские активности;
- мастер-классы;
- лекции;
- конкурсы;
- открытые мероприятия без лимита гостей.

Из этого следует важное требование: у события должен быть режим:
- с ограничением мест;
- без ограничения мест;
- по заявке с подтверждением организатора (можно добавить позже).

6. Бизнес-логика

1) Организатор создает событие.
2) Событие проходит модерацию или публикуется сразу.
3) Событие появляется на общей доске.
4) Студент открывает карточку и записывается.
5) Если лимит достигнут:
- запись блокируется;
- либо формируется лист ожидания.

7. Структура разделов сайта

Предлагаемая структура:
- Главная / Доска событий
- Категории
- Карточка события
- Мои записи
- Мои мероприятия
- Создать событие
- Админ-панель
- О сервисе / Помощь

8. Фильтры и поиск

Чтобы сервис был удобным, нужны:
- поиск по названию;
- фильтр по категории;
- фильтр по дате;
- фильтр по формату;
- фильтр по месту;
- фильтр по наличию свободных мест;
- фильтр "только актуальные".

9. Данные, которые хранит система

Основные сущности:

Пользователь:
- id;
- ФИО;
- email;
- роль;
- факультет / группа;
- телефон.

Событие:
- id;
- название;
- описание;
- категория;
- дата и время;
- место;
- формат;
- контакты;
- изображение;
- лимит мест;
- статус;
- id организатора.

Запись на событие:
- id;
- id студента;
- id события;
- дата записи;
- статус участия.

Категория:
- id;
- название.

10. Нефункциональные требования

- адаптивность под телефон и ПК;
- простота интерфейса;
- защита персональных данных;
- быстрая загрузка страниц;
- базовая отказоустойчивость;
- логирование действий;
- безопасность авторизации.

11. Технологический план

Для дипломного проекта подойдет такой стек:

Frontend:
- React / Next.js

Backend:
- Node.js + NestJS или Express

Database:
- PostgreSQL

Auth:
- JWT или session-based auth

Storage:
- локальное хранилище или S3-совместимое для изображений

Deployment:
- Docker
- Nginx
- VPS / university server

Если нужен стек попроще для диплома:
- React + Node.js + PostgreSQL

12. Этапы разработки

1) Аналитика
- собрать требования;
- описать роли;
- описать пользовательские сценарии;
- выделить MVP.

2) Проектирование
- сделать use case diagram;
- продумать ER-диаграмму базы данных;
- спроектировать API;
- подготовить wireframes интерфейса.

3) Дизайн
- макеты страниц;
- UI-kit;
- адаптивная версия.

4) Backend-разработка
- авторизация;
- CRUD событий;
- запись на событие;
- роли и права;
- админские функции.

5) Frontend-разработка
- доска событий;
- карточка события;
- формы входа и регистрации;
- кабинет студента;
- кабинет организатора.

6) Тестирование
- функциональное;
- UI-тестирование;
- проверка ролей и прав доступа;
- тесты записи на событие при лимитах.

7) Развертывание
- публикация на сервере;
- подключение БД;
- настройка домена и HTTPS.

8) Документация
- пояснительная записка;
- описание архитектуры;
- руководство пользователя;
- руководство администратора.

13. Что можно включить в диплом как расширенный функционал

После MVP можно добавить:
- уведомления на email;
- push-уведомления;
- карту с местом проведения;
- QR-регистрацию на входе;
- отзывы о мероприятиях;
- рейтинг организаторов;
- рекомендации событий;
- интеграцию с расписанием университета;
- импорт в календарь;
- аналитику по вовлеченности студентов.

14. Возможные риски

- сложность модерации контента;
- дублирование мероприятий;
- недостоверные контакты;
- перегруженный интерфейс;
- проблемы с ролями доступа;
- необходимость защиты персональных данных.

15. Результат проекта

На выходе должен получиться сервис, который:
- централизует всю внеучебную активность университета;
- упрощает публикацию и поиск мероприятий;
- делает запись студентов удобной;
- повышает вовлеченность студентов в жизнь университета.
