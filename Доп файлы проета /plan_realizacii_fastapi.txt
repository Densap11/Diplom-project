План реализации веб-сервиса агрегации внеучебной деятельности университета на FastAPI

1. Цель этапа реализации

Разработать рабочий веб-сервис, в котором:
- организаторы создают и публикуют события;
- студенты просматривают общую доску событий;
- студенты открывают карточку мероприятия и записываются;
- администратор управляет контентом и ролями.

2. Выбранный стек

Backend:
- FastAPI
- SQLAlchemy 2.0
- Alembic
- Pydantic
- PostgreSQL
- JWT-аутентификация

Frontend:
- Next.js или React

Инфраструктура:
- Docker
- Docker Compose
- Nginx

3. Общая стратегия реализации

Проект нужно делать поэтапно, начиная с MVP:
- сначала подготовить архитектуру и базу данных;
- затем реализовать авторизацию и роли;
- после этого создать модуль событий;
- затем реализовать запись студентов на события;
- потом добавить личные кабинеты;
- в конце выполнить тестирование, полировку и деплой.

4. Этап 1. Подготовка проекта

Цель:
Создать базовый каркас backend-проекта на FastAPI.

Задачи:
- создать структуру проекта;
- настроить виртуальное окружение;
- подключить FastAPI;
- подключить PostgreSQL;
- настроить файл конфигурации `.env`;
- подключить SQLAlchemy и Alembic;
- подготовить Docker-конфигурацию.

Результат:
- сервер запускается;
- есть подключение к базе данных;
- настроены миграции;
- проект готов к дальнейшей разработке.

5. Этап 2. Проектирование базы данных

Цель:
Спроектировать сущности и связи.

Основные таблицы:
- users
- events
- event_registrations
- categories

Дополнительно можно предусмотреть:
- event_images
- audit_logs
- notifications

Что нужно определить:
- поля таблиц;
- внешние ключи;
- ограничения уникальности;
- роли пользователей;
- статусы событий и регистраций.

Результат:
- готова ER-модель;
- созданы SQLAlchemy-модели;
- выполнена первая миграция.

6. Этап 3. Модуль пользователей и авторизации

Цель:
Реализовать регистрацию, вход и разграничение прав.

Задачи:
- регистрация пользователя;
- вход по email и паролю;
- хеширование пароля;
- выдача JWT-токена;
- получение текущего пользователя;
- реализация ролей: student, organizer, admin;
- ограничение доступа к защищенным маршрутам.

Результат:
- пользователь может зарегистрироваться и войти;
- API умеет различать роли;
- защищенные endpoints работают корректно.

7. Этап 4. Модуль категорий

Цель:
Создать справочник категорий мероприятий.

Задачи:
- CRUD для категорий;
- получение списка категорий;
- ограничение редактирования для администратора.

Результат:
- категории можно использовать при создании событий;
- фильтрация по категориям становится возможной.

8. Этап 5. Модуль событий

Цель:
Реализовать основной функционал публикации мероприятий.

Задачи:
- создание события организатором;
- редактирование события;
- удаление события;
- публикация и снятие с публикации;
- получение списка событий;
- получение карточки одного события;
- фильтрация по дате, категории, формату и статусу;
- поддержка лимита мест и безлимитного формата.

Основные поля события:
- title
- short_description
- full_description
- category_id
- event_date
- location
- format
- contacts
- max_participants
- is_unlimited
- status
- organizer_id

Результат:
- организатор публикует события;
- студенты видят общую доску мероприятий;
- работает карточка события.

9. Этап 6. Модуль записи на событие

Цель:
Добавить механизм регистрации студентов.

Задачи:
- запись студента на событие;
- отмена записи;
- проверка, что пользователь не записывается дважды;
- проверка лимита мест;
- возврат количества свободных мест;
- список участников для организатора.

Дополнительно:
- предусмотреть статус waiting_list на будущее;
- предусмотреть статус cancelled.

Результат:
- студент может записаться на мероприятие;
- организатор видит список записавшихся;
- система контролирует лимиты.

10. Этап 7. Личные кабинеты

Цель:
Сделать API для персональных разделов пользователей.

Для студента:
- мои записи;
- мои предстоящие события;
- отмена участия.

Для организатора:
- мои события;
- участники по каждому событию;
- статистика по заполненности.

Для администратора:
- список всех событий;
- список пользователей;
- управление статусами событий.

Результат:
- у каждой роли появляется свой рабочий кабинет.

11. Этап 8. Администрирование и модерация

Цель:
Обеспечить контроль качества контента.

Задачи:
- просмотр всех созданных событий;
- смена статуса события;
- скрытие нежелательных публикаций;
- управление ролями пользователей;
- журнал административных действий.

Результат:
- система пригодна для использования внутри университета.

12. Этап 9. Документирование API

Цель:
Сделать API понятным для фронтенда и для демонстрации диплома.

Задачи:
- описать схемы запросов и ответов;
- поддерживать чистые Pydantic-схемы;
- использовать Swagger/OpenAPI;
- добавить примеры запросов.

Результат:
- API легко тестировать;
- фронтенд-разработка ускоряется;
- дипломный проект выглядит профессионально.

13. Этап 10. Тестирование

Цель:
Проверить корректность ключевой логики.

Что тестировать:
- регистрацию и вход;
- права доступа;
- создание события;
- редактирование и удаление события;
- запись на событие;
- поведение при достижении лимита;
- отмену записи;
- получение списков событий.

Инструменты:
- pytest
- httpx / TestClient

Результат:
- критическая бизнес-логика покрыта тестами;
- снижается риск ошибок на защите проекта.

14. Этап 11. Frontend-интеграция

Цель:
Подключить клиентскую часть к API.

Минимальные страницы:
- главная доска событий;
- страница события;
- вход и регистрация;
- кабинет студента;
- кабинет организатора;
- форма создания события.

Результат:
- сервис становится полностью usable для демонстрации.

15. Этап 12. Деплой и подготовка к защите

Цель:
Развернуть проект и подготовить его к презентации.

Задачи:
- собрать контейнеры;
- настроить PostgreSQL;
- настроить Nginx;
- подключить домен или IP;
- настроить HTTPS при необходимости;
- подготовить тестовые данные;
- оформить README.

Результат:
- проект можно запустить и показать комиссии.

16. Приоритет разработки MVP

Рекомендуемая очередность:

1) каркас FastAPI-проекта
2) база данных и миграции
3) пользователи и JWT-аутентификация
4) роли и права доступа
5) категории
6) события
7) запись на события
8) личные кабинеты
9) тестирование
10) подключение фронтенда

17. Предлагаемая структура backend-проекта

app/
- main.py
- core/
- db/
- models/
- schemas/
- api/
- services/
- repositories/
- dependencies/
- utils/
- tests/

Внутри можно разнести модули так:
- api/v1/auth.py
- api/v1/users.py
- api/v1/events.py
- api/v1/categories.py
- api/v1/registrations.py

18. Что нужно реализовать в первую очередь прямо сейчас

Первый практический шаг:
- создать backend-каркас;
- настроить зависимости;
- подключить базу данных;
- создать модели User, Event, Category, EventRegistration;
- подготовить миграции;
- сделать базовый endpoint проверки работоспособности.

19. Итог реализации

После выполнения плана получится backend-система, которая:
- поддерживает роли студентов, организаторов и администраторов;
- позволяет публиковать и просматривать события;
- позволяет записываться на мероприятия;
- предоставляет основу для дальнейшего расширения сервиса.
